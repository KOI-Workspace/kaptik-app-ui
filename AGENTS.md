# Kaptik 공통 에이전트 작업 지침

이 문서는 AI 개발 에이전트가 Kaptik의 서비스 목적과 이 저장소의 역할을 이해하고, 기획 의도가 개발자에게 정확히 전달되는 화면을 만들기 위한 공통 작업 기준이다.

## 1. 서비스 목적

- Kaptik은 글로벌 K-pop 팬을 위한 고품질 다국어 자막 서비스다.
- 언어 장벽 때문에 아티스트의 라이브, 영상, 말투, 농담과 문화적 맥락을 충분히 이해하지 못하는 팬의 문제를 해결한다.
- 단순한 문장 번역을 넘어 화자 구분, 자연스러운 표현, 시간 동기화, 문화 맥락 설명까지 제공하는 것이 핵심 가치다.
- 사용자는 좋아하는 아티스트를 팔로우하고, 라이브 또는 영상 링크를 통해 원하는 언어의 자막을 이용할 수 있다.

## 2. 이 저장소의 역할

- 이 저장소는 최종 서비스나 실제 백엔드가 아니라, 기획자가 앱 개발자에게 제품 기획과 화면을 전달하기 위한 실행 가능한 UI 프로토타입이다.
- 개발자는 이 화면을 통해 정보 구조, 사용자 흐름, 화면 상태, 인터랙션, 디자인 방향을 이해한다.
- 실제 서버 기능보다 다음 내용을 명확하게 보여주는 것이 중요하다.
  - 사용자가 어떤 순서로 서비스를 이용하는지
  - 버튼을 눌렀을 때 어떤 화면과 상태가 나타나는지
  - 로그인과 요금제에 따라 무엇이 달라지는지
  - 자막과 번역 경험이 실제 앱에서 어떻게 보이고 작동하는지
  - 로딩, 오류, 빈 상태, 선택 상태처럼 개발자가 놓치기 쉬운 경우
- 목업 기능은 실제 기능처럼 오해되지 않도록 코드와 문서에서 구분한다.
- 백엔드 구조를 임의로 확정하기보다 화면 동작과 필요한 데이터가 무엇인지 분명하게 표현한다.

현재 프로토타입은 HTML, CSS, Vanilla JavaScript ES Module로 구성된 모바일 웹 앱이다. 해시 기반 SPA 라우터를 사용하며 인증, 결제, 번역은 정적 데이터와 `localStorage` 기반 목업이다. 기준 화면 폭은 390px, 최대 앱 폭은 430px이다.

## 3. 제품 원칙

- 자막 품질과 이해도를 기능 수보다 우선한다.
- 번역문은 기계적인 직역보다 글로벌 팬이 자연스럽게 이해할 수 있어야 한다.
- 아티스트별 화자를 쉽게 구분할 수 있어야 한다.
- 자막은 영상 시간과 연결되며, 자막을 눌러 해당 장면으로 이동할 수 있어야 한다.
- 말장난, 팬덤 표현, 한국 문화처럼 번역만으로 어려운 내용은 맥락 설명을 제공한다.
- 글로벌 사용자를 전제로 언어 변경과 긴 번역 문구에서도 레이아웃이 깨지지 않아야 한다.
- 로그인과 결제 유도는 이용 흐름을 방해하지 않으면서 조건을 분명히 전달해야 한다.
- K-pop 팬에게 친숙한 감성을 사용하되 특정 플랫폼의 브랜드나 UI를 그대로 복제하지 않는다.

## 4. 현재 진행 상황

구현된 주요 흐름:

- Home / Translate / My 하단 탭과 해시 라우팅
- 신규·기존 사용자를 구분하는 온보딩
- Google 데모 로그인과 이메일 회원가입·로그인
- 필수 약관 및 선택 마케팅 동의
- 아티스트 선택, 팔로우, 라이브 알림
- Free / Basic / Pro 요금제와 접근 게이트
- URL 입력 후 영상을 즉시 재생하고 자막 영역에서 생성 상태를 안내하는 흐름
- YouTube 또는 가상 재생 클럭 기반 플레이어
- 전체 자막 탐색, 현재 자막 강조, 자막 클릭 이동, 언어 변경, 문화 맥락 설명
- 한국어 포함 10개 UI 언어
- 모바일 다크 UI, 모달, 토스트, 플로팅 하단 내비게이션

실제 앱 개발 단계에서 연결해야 할 기능:

- 실제 OAuth 및 서버 인증
- 안전한 계정 저장과 세션 관리
- 실제 결제와 구독 관리
- 실제 URL 분석, STT, 번역 API
- Weverse 등 외부 플랫폼의 실제 영상 재생 방식
- 백엔드 데이터 동기화와 푸시 알림

현재 우선 확인할 문제:

1. Home은 Player에 `{ feed }`를 전달하지만 Player는 `params.source`를 읽어 선택한 영상 대신 기본 데모가 열릴 수 있다.
2. Translate는 URL 형식과 빈 값 검증 없이 처리 흐름을 시작한다.
3. 로그인 화면과 게이트 로그인 모달의 회원가입 및 약관 처리 방식이 다르다.
4. 자동화 테스트와 린트 설정이 없어 변경 후 브라우저 검증이 필수다.

## 5. 확정된 Translate와 Player UX

다음 내용은 현재 기획에서 확정된 동작이다. 관련 화면을 수정할 때 임의로 이전 방식으로 되돌리지 않는다.

### Translate 진입과 자막 생성

- 영상 링크 제출 후 전체 화면 `Generating` 또는 처리 오버레이를 띄우지 않는다.
- 로그인과 Pro 접근 게이트를 통과하면 Player로 즉시 이동해 영상을 먼저 재생한다.
- 자막 생성 중에는 영상 시청을 막지 않고 자막 바텀시트 안에서만 작은 준비 상태를 보여준다.
- 준비 상태는 `자막을 준비하고 있어요`, `영상은 계속 시청할 수 있어요`처럼 현재 상황과 이용 가능한 행동을 함께 알린다.
- 자막이 준비되면 완료 안내를 짧게 보여준 뒤 전체 자막 목록으로 자연스럽게 전환한다.
- YouTube처럼 스크립트 추출이 가능한 플랫폼과 STT가 필요한 플랫폼은 준비 시간이 다를 수 있지만 화면 흐름은 동일하다.

### 자막 목록과 시간 동기화

- 자막 생성이 끝나면 현재 시점까지의 자막만 추가하는 방식이 아니라 영상 전체 자막을 한 번에 표시한다.
- 자막 목록에서는 미래 또는 새로운 자막이 위에 있고, 과거 자막이 아래에 있다.
- 사용자는 재생 시점과 관계없이 스크롤해 이전 자막과 이후 자막을 모두 볼 수 있어야 한다.
- 영상 시간이 지나도 자막 항목을 새로 추가하거나 목록 순서와 사용자의 스크롤 위치를 자동으로 변경하지 않는다.
- 현재 재생 구간의 자막은 브랜드 보라색 배경으로 지속해서 강조한다.
- 자막을 누르면 영상만 해당 타임스탬프로 이동하고, 누른 자막의 위치나 목록 스크롤은 바뀌지 않는다.
- 타임스탬프 이동 후에는 이동한 구간의 자막이 현재 자막 강조 상태가 된다.
- 언어 변경으로 자막을 다시 렌더링할 때도 가능한 한 기존 스크롤 위치를 유지한다.

### 결제 바텀시트

- 링크 번역은 Pro 기능이다.
- Free 사용자가 진입하면 Basic과 Pro를 모두 보여준다.
- 이미 Basic을 이용 중인 사용자가 링크 번역을 시도하면 Pro만 보여준다.
- 바텀시트 하단의 `Start Pro`와 `Later` 버튼은 사용하지 않는다.
- 각 가격 카드는 전체 영역이 결제 진입 버튼이며, 테두리·배경·화살표·누름 상태로 클릭 가능함을 분명히 표현한다.
- 닫기는 우측 상단 X 또는 바깥 영역을 사용한다.

## 6. 프로토타입 작업 원칙

- 화면을 수정할 때 먼저 “개발자에게 어떤 기획 의도를 전달해야 하는가”를 판단한다.
- 실제 앱에서 필요한 핵심 상태를 프로토타입에서도 눌러보고 확인할 수 있게 만든다.
- 보기 좋은 정적 화면보다 주요 버튼과 화면 전환이 이어지는 시연 가능한 흐름을 우선한다.
- 실제 API가 없어도 성공, 실패, 로딩, 권한 부족 상태를 이해할 수 있는 목업을 제공한다.
- 데모 데이터는 Kaptik의 자막 품질과 사용 경험을 설명할 만큼 구체적으로 작성한다.
- 구현이 확정되지 않은 부분은 과도하게 가정하지 않고 목업임을 명시한다.
- 기획과 관계없는 서버 구조, 관리자 기능, 범용 프레임워크를 임의로 추가하지 않는다.
- 화면별 용어, 요금제 조건, 사용자 상태가 서로 충돌하지 않게 유지한다.
- 사소한 디자인 수정에는 스킬을 습관적으로 사용하지 않는다.
- 신규 화면, 대규모 UI 개편, 전문 산출물처럼 실질적인 도움이 있을 때만 필요한 최소 스킬을 사용한다.

## 7. 주요 파일

- `index.html`: 앱 루트와 외부 스크립트
- `js/app.js`: 앱 부팅, 라우트 등록, 하단 내비게이션
- `js/router.js`: 해시 라우터와 화면 정리
- `js/state.js`: 전역 상태와 `localStorage`
- `js/auth.js`: 목업 회원가입·로그인 검증
- `js/gate.js`: 로그인 및 요금제 접근 제어
- `js/modals.js`: 로그인, 결제, 팔로우, 약관, 토스트
- `js/views/`: 화면별 렌더 함수
- `js/data.js`: 아티스트, 라이브, 자막 데모 데이터
- `js/strings.js`: 다국어 UI 문자열
- `js/i18n.js`: 번역 조회와 언어 변경
- `css/tokens.css`: 디자인 토큰
- `css/app.css`: 공통 및 일반 화면 스타일
- `css/player.css`: 플레이어 전용 스타일
- `design.md`, `design.json`: 디자인 기준
- `spec.html`: 초기 UI 참고 명세

## 8. 아키텍처 규칙

- 화면은 `renderX(params, root)` 형태로 작성하고 `app.js`에 등록한다.
- 화면에서 만든 전역 이벤트, 타이머, 플레이어는 정리 함수를 반환해 이탈 시 제거한다.
- 화면 이동은 `navigate(route, params)`를 사용한다.
- 전역 상태는 `state.js`를 단일 진실 소스로 사용한다.
- 상태 변경은 `setState`, `setPlan`, 팔로우 관련 함수를 사용한다.
- 접근 제한 기능은 `requireAccess`를 통과시킨다.
- UI 문구는 `strings.js`에 키를 추가하고 `t()`로 사용한다.
- 공통 모달은 `modal-root`, 토스트는 `toast-root`에 렌더한다.
- 사용자 입력을 HTML에 넣을 때는 반드시 이스케이프한다.

## 9. 코드 작성 규칙

- 들여쓰기는 2칸을 사용한다.
- 변수와 함수는 `camelCase`, 컴포넌트성 이름은 `PascalCase`를 사용한다.
- 코드 주석과 문서는 한국어로 작성한다.
- 기존 Vanilla JavaScript 구조를 유지하고 요청 없이 프레임워크나 빌드 도구를 도입하지 않는다.
- 복잡한 함수나 공개 유틸에는 필요한 경우 짧은 한국어 JSDoc을 추가한다.
- CSS 값은 가능한 한 `css/tokens.css` 변수를 사용한다.
- 인라인 스타일은 동적 값 등 불가피한 경우 외에는 피한다.
- 터치 대상은 최소 44px, 본문 대비는 4.5:1을 지향한다.
- 사용자 변경과 무관한 리팩터링은 하지 않는다.

## 10. 디자인 기준

- 배경은 `#050505`, 카드 계열은 `#191919`과 `#202020`을 기본으로 한다.
- 브랜드 보라색 `#8353EB`은 CTA, 선택 상태, AI·자막 강조에 제한적으로 사용한다.
- 모바일 우선으로 작업하고 390px 화면을 먼저 확인한다.
- 개발자가 규격을 이해할 수 있도록 간격, 크기, 선택 상태를 일관되게 유지한다.
- 하단 내비게이션과 플레이어 FAB가 콘텐츠를 가리지 않게 safe area를 고려한다.
- 디자인 변경 전 `design.md`와 `design.json`을 확인한다.
- Weverse 로고나 외부 서비스 UI를 그대로 복제하지 않는다.

## 11. 상태와 데이터 계약

주요 앱 상태:

```js
{
  isLoggedIn,
  onboardingDone,
  follows,
  agreements,
  plan,
  isPaid,
  uiLang,
  defaultLang,
  user
}
```

- `plan`은 `free | basic | pro` 중 하나다.
- `isPaid`는 Pro 접근용 파생 값이며 `setPlan()`으로 동기화한다.
- 라이브 시청은 Basic 이상, 링크 번역은 Pro가 필요하다.
- 상태 저장 키는 `kaptik.state.v1`, 목업 계정 키는 `kaptik.accounts.v1`이다.
- 저장 구조 변경 시 기존 데이터와의 호환 또는 마이그레이션을 고려한다.

## 12. 실행과 검증

루트에서 정적 서버를 실행한다.

```bash
python3 -m http.server 8000
```

브라우저에서 `http://127.0.0.1:8000`을 연다. `file://`로 열면 ES Module 제약으로 정상 동작하지 않을 수 있다.

JavaScript 문법 확인:

```bash
for file in js/*.js js/views/*.js; do
  node --input-type=module --check < "$file" || exit 1
done
```

변경 후 최소 확인 항목:

1. 첫 방문 온보딩과 기존 사용자 분기
2. Google 로그인, 이메일 가입·로그인, 오류 메시지
3. 약관 필수 동의와 마케팅 선택 동의
4. Home 라이브 진입 시 로그인·요금제 게이트
5. Translate의 Pro 게이트, 영상 즉시 재생, 자막 영역 준비 상태
6. Player 전체 자막 표시, 현재 자막 강조, 스크롤 유지, seek, 언어 변경, 이탈 시 정리
7. My의 로그인, 요금제, 팔로우, 언어, 약관 설정
8. 390px 모바일 화면과 데스크톱 중앙 정렬
9. 브라우저 콘솔 오류와 남은 타이머·이벤트 리스너

화면 변경 시 추가 확인:

1. 개발자가 별도 설명 없이 화면 목적과 다음 행동을 이해할 수 있는가
2. 기본, 선택, 로딩, 성공, 실패, 빈 상태가 필요한 만큼 표현되어 있는가
3. 실제 기능과 목업 기능이 혼동되지 않는가
4. 한국어 외 긴 문구에서도 버튼과 레이아웃이 유지되는가
5. 화면 간 용어, 요금제 조건, 사용자 상태가 일관적인가

## 13. 작업 우선순위

1. 앱 부팅을 막는 문법 오류와 핵심 사용자 흐름 오류
2. 개발자에게 전달해야 할 핵심 사용자 흐름과 화면 상태
3. 화면 간 데이터 계약, 용어, 상태의 일관성
4. 자막 품질을 보여주는 플레이어와 문화 맥락 경험
5. 글로벌 언어 대응, 입력 검증, 접근성, 오류 처리
6. 시각적 완성도와 실제 API 연결을 위한 요구사항 정리
7. 테스트와 개발 도구 구성

## 14. 작업 보고 방식

- 변경 이유를 짧게 설명한다.
- 오류는 `증상 → 원인 → 해결 방법` 순서로 설명한다.
- 실행한 검증과 실행하지 못한 검증을 구분한다.
- 프로토타입 동작을 실제 서버 기능처럼 표현하지 않는다.
- 화면 변경이 어떤 기획 의도를 개발자에게 전달하는지 설명한다.
- 중요한 미해결 문제는 파일과 영향 범위를 함께 남긴다.
